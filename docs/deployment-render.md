# Deploying DocPilot on Render

This runbook describes DocPilot's supported production deployment. It is an
operator guide, not evidence that the repository has already been deployed.

## Platform Decision

Render is the selected target because its Blueprint format can describe both
existing Docker services, managed PostgreSQL, a migration hook, health checks,
secrets, and durable upload storage without adding Kubernetes, Terraform, a
container registry, or a second deployment system.

Fly.io was also considered. It supports Docker services, release commands,
volumes, and managed PostgreSQL, but this repository would need separate app
configuration and lifecycle management for the frontend and backend. A plain VPS
would require the project owner to operate PostgreSQL, TLS, backups, and host
patching. Those options add operational work without improving this milestone.

Relevant platform documentation:

- [Render Blueprints](https://render.com/docs/infrastructure-as-code)
- [Render Blueprint specification](https://render.com/docs/blueprint-spec)
- [Render Docker services](https://render.com/docs/docker)
- [Render persistent disks](https://render.com/docs/disks)
- [Render PostgreSQL](https://render.com/docs/postgresql)

## Production Architecture

```text
Browser
  | HTTPS
  v
Render frontend web service (Nginx + static HTML/CSS/JavaScript)
  | HTTPS API requests
  v
Render backend web service (FastAPI/Uvicorn)
  | private network                 | filesystem
  v                                 v
Managed PostgreSQL 16         Persistent disk at /app/uploads
```

The database stores users, document metadata, extracted text, chunks,
conversations, and evaluation records. Uploaded PDF and DOCX source files live on
the persistent disk. Both stores are required for a complete recovery.

## Resources and Cost Awareness

The root `render.yaml` requests:

- One Starter backend web service
- One Starter frontend web service
- One Basic-256mb PostgreSQL database
- One 1 GB persistent disk attached to the backend

Render pricing and plan names can change. Review the estimated monthly total in
the Render dashboard before approving the Blueprint. The backend cannot use a
free service while it has a persistent disk. A lower-cost static-site frontend
would stop using the repository's frontend Docker image, so it is intentionally
outside this milestone.

The Blueprint uses the Frankfurt region. Change all three resource regions
together before the first deployment if another region is required. Moving an
existing database or persistent disk later is a data-migration operation, not a
routine configuration edit.

## Prerequisites

Before deploying:

1. Merge the deployment branch into `main` and confirm GitHub CI passed on the
   merged commit.
2. Create or sign in to a Render account with billing enabled.
3. Connect Render to the GitHub repository with read access to the repository.
4. Create an OpenAI API key for the production environment and set provider-side
   usage limits appropriate for the application.
5. Decide the public service names or custom domains.

Do not reuse local `.env` values or a development JWT secret.

## First Deployment

1. In Render, choose **New > Blueprint** and select the DocPilot repository.
2. Render detects `render.yaml`. Review every proposed resource and its cost.
3. Supply the prompted values:
   - `OPENAI_API_KEY`: production provider secret.
   - Backend `DOCPILOT_CORS_ORIGINS`: the frontend HTTPS origin, for example
     `https://docpilot-frontend.onrender.com`. Do not include a trailing slash.
   - Frontend `DOCPILOT_API_URL`: the backend HTTPS URL, for example
     `https://docpilot-backend.onrender.com`. Do not include a trailing slash.
4. Approve the Blueprint and wait for the database and both services to finish.
5. Compare the actual service URLs shown by Render with the two configured URL
   values. If Render assigned a different hostname, update both environment
   variables and redeploy the affected services.
6. Confirm the backend pre-deploy log shows `alembic upgrade head` completed
   before Uvicorn started.
7. Complete the verification checklist below.

The service definitions use `autoDeployTrigger: checksPass`, so later commits to
`main` deploy only after the associated GitHub checks succeed. Render does not
build or deploy pull-request commits through this production Blueprint.

## Environment Variables and Secrets

| Variable | Service | Source | Purpose |
| --- | --- | --- | --- |
| `DATABASE_URL` | Backend | Managed database connection string | SQLAlchemy and Alembic connection; normalized to psycopg v3 at runtime |
| `JWT_SECRET_KEY` | Backend | Generated by Render | Signs access tokens; changing it invalidates existing sessions |
| `OPENAI_API_KEY` | Backend | Render secret prompt | Embeddings and grounded answers; never put it in Git |
| `DOCPILOT_CORS_ORIGINS` | Backend | Operator-supplied | Comma-separated exact browser origins allowed to call the API |
| `DOCPILOT_ENVIRONMENT` | Backend | Blueprint value `production` | Enables fail-closed production validation and disables API docs |
| `DOCPILOT_TRUSTED_HOSTS` | Backend | Optional operator value | Extra exact custom-domain hostnames; Render's assigned hostname is detected automatically |
| `DOCPILOT_RATE_LIMIT_ENABLED` | Backend | Blueprint value `true` | Enables bounded per-process auth, upload, and AI request limits |
| `DOCPILOT_UPLOAD_DIR` | Backend | `/app/uploads` | Persistent source-document directory |
| `DOCPILOT_API_URL` | Frontend | Operator-supplied | Public HTTPS backend base URL written to runtime `config.js` |
| `PORT` | Backend | Injected by Render | Uvicorn bind port; local fallback remains `8000` |
| `PORT` | Frontend | Blueprint value `8080` | Unprivileged Nginx listener port |

The AI model, embedding, retrieval, evaluation, and conversation-limit variables
are declared in `render.yaml` with the same defaults as local Docker Compose.
Adjust them in the Render dashboard when needed. Environment values managed in
the dashboard should be treated as production configuration and reviewed before
redeployment.

Only exact HTTPS frontend origins should be present in
`DOCPILOT_CORS_ORIGINS`. For multiple domains, use a comma-separated value such
as `https://app.example.com,https://www.example.com`. Production startup rejects
wildcards, opaque `null` origins, short/default JWT secrets, and wildcard or
missing trusted hosts. Render's `RENDER_EXTERNAL_HOSTNAME` is automatically
trusted; add exact custom domains through `DOCPILOT_TRUSTED_HOSTS` when needed.

## Database Initialization and Migrations

Render runs this backend pre-deploy command:

```bash
alembic upgrade head
```

On a fresh managed database, the reviewed baseline migration creates all current
tables before the new backend instance starts. A failed migration stops that
backend deployment. The runtime application does not call
`Base.metadata.create_all()`; isolated pytest fixtures remain the only users of
`create_all()`.

For an existing database created before Alembic, do not point the Blueprint at it
and run the baseline blindly. Back it up, compare the schema with the baseline,
and follow the stamping procedure in the main README before enabling automatic
deployments.

Future migrations should be backward-compatible with the previous application
version whenever possible. A Render code rollback does not automatically reverse
a database migration.

## Uploaded-File Persistence

The backend disk is mounted at `/app/uploads`, matching
`DOCPILOT_UPLOAD_DIR`. Uploaded source files therefore survive normal restarts,
container replacement, and application redeployment.

Important limitations of this design:

- A persistent disk belongs to one backend service and prevents horizontal
  scaling to multiple instances.
- Disk-backed deploys can have a brief service interruption rather than
  zero-downtime replacement.
- The disk is not available during image builds or pre-deploy commands. DocPilot's
  pre-deploy command only migrates PostgreSQL and does not require uploaded files.
- This milestone does not copy uploads to object storage or another region.

If scale-out, cross-region recovery, direct upload, or CDN delivery becomes a
requirement, replace the filesystem repository with object storage in a separate,
tested application milestone.

## HTTPS, Health Checks, and Service URLs

Render terminates HTTPS for each public web service. Keep application URLs on
HTTPS and configure custom-domain DNS and certificates through Render if a custom
domain is used.

Expected endpoints after deployment:

- Frontend: `https://<frontend-service-host>/`
- Backend: `https://<backend-service-host>/`
- API documentation: `https://<backend-service-host>/docs`
- Backend health: `https://<backend-service-host>/health`

The backend health check verifies that FastAPI responds. The frontend check
verifies that Nginx serves the application root. These are liveness checks; the
current `/health` handler does not perform a database or OpenAI provider query.

## Deployment Verification

After the first deploy and every material platform change:

1. Confirm both Render services report healthy.
2. Open `/health` on the backend and verify `{"status":"healthy"}`.
3. Open the frontend and confirm browser requests target the production backend,
   not `127.0.0.1`.
4. Register a temporary user and sign in.
5. Upload one small PDF or DOCX and confirm it appears in the document list.
6. Ask one grounded question, confirming an authorized production OpenAI request
   is expected and citations are returned when relevant.
7. Restart or redeploy the backend and confirm the user, document record, and
   uploaded source file still exist.
8. Delete the temporary document and account data as appropriate.
9. Review migration, backend, frontend, and database logs for errors.

Do not perform step 6 against an unapproved provider account, and do not use
sensitive documents for smoke testing.

## Redeployment and Rollback

Normal redeployment is automatic after checks pass on `main`. For a manual
redeploy, select the service in Render and deploy the desired `main` commit. The
backend migration completes before the backend release; deploy the frontend after
the backend when a change requires both services.

For application rollback:

1. Identify the last healthy deploy and review migrations introduced since it.
2. Back up PostgreSQL and the upload disk before changing schema state.
3. Roll back the backend and frontend to compatible commits in the Render
   dashboard.
4. Do not run `alembic downgrade` unless that specific downgrade has been reviewed
   and tested against a restored backup. The baseline downgrade drops all tables
   and is destructive.
5. Re-run the verification checklist.

## Backup and Restore Considerations

Use Render's database backup/recovery features at a retention level that matches
the project's recovery requirements. Periodically test a restore into a separate
database rather than assuming that a listed backup is usable.

Render persistent disks have platform snapshots, but retention and restore
options depend on current platform behavior and plan. Verify those settings in
the dashboard and keep an independent export of important uploads if the recovery
objective exceeds the platform guarantee.

Coordinate database and upload backups. Restoring only PostgreSQL can leave
document records whose source files are missing; restoring only the disk can leave
orphaned files. Record backup timestamps and restore the closest consistent pair.

## Troubleshooting

- **Backend build succeeds but never becomes healthy:** inspect pre-deploy and
  service logs, then verify `DATABASE_URL`, the migration revision, and Render's
  injected `PORT`.
- **Browser reports a CORS error:** compare the browser's exact frontend origin
  with `DOCPILOT_CORS_ORIGINS`; remove trailing slashes and redeploy the backend.
- **Frontend calls localhost:** verify `DOCPILOT_API_URL`, redeploy the frontend,
  and inspect `/config.js` in the browser.
- **Upload fails with a permission error:** verify the disk is attached at
  `/app/uploads` and `DOCPILOT_UPLOAD_DIR` has the same value. Do not replace the
  disk until its data has been backed up.
- **AI features return configuration errors:** confirm `OPENAI_API_KEY` is present
  in the backend only and review provider usage limits. Authentication, document
  listing, and other non-AI endpoints should remain independent.
- **Migration fails because tables already exist:** stop automatic deployment and
  use the reviewed existing-database adoption procedure; do not delete tables or
  stamp an unverified schema.

## Known Deployment Limitations

- No production Render resources or public URLs are created by this repository
  alone; an authorized account owner must apply the Blueprint.
- Upload persistence uses a single attached disk, so the backend is single-instance.
- There is no cross-region failover or object-storage replication.
- The health endpoint is shallow and does not prove database or AI availability.
- Log aggregation, alerting, error tracking, and deeper operational metrics belong
  to the later production-monitoring milestone.
- Rate limiting is intentionally per backend process. A future multi-instance
  deployment needs a shared edge or datastore-backed limiter.
